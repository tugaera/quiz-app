## TIMER ARCHITECTURE (Source of Truth: Server)
The timer must be authoritative and server-driven. Clients MUST NOT manage
their own independent countdowns.

### How it works:
When the quiz starts or advances to the next question, the server:
1. Records `current_question_index` and `question_started_at: timestamptz`
   in the `quiz_sessions` table
2. Broadcasts a realtime event with:
   {
     questionIndex: number,
     questionStartedAt: string,   // ISO timestamp
     timeLimitSeconds: number
   }

Clients compute remaining time as:
  const elapsed = (Date.now() - new Date(questionStartedAt).getTime()) / 1000
  const remaining = Math.max(0, timeLimitSeconds - elapsed)

This means:
- All clients share the same deadline regardless of when they connected
- A user who closes and reopens the browser fetches current session state
  on mount and resumes mid-timer with the correct remaining time
- A late joiner gets the same treatment: they join partway through a
  question with reduced time, which is correct and fair

### Server-side auto-progression:
Use a Supabase Edge Function triggered on a schedule (or pg_cron) that:
- Checks active quiz_sessions where
  question_started_at + time_limit_seconds <= now()
- Locks answers for the current question
- If more questions remain:
  - Increments current_question_index
  - Sets new question_started_at to now()
  - Broadcasts next_question event
- If no more questions:
  - Sets session status to 'ended'
  - Broadcasts quiz_end event

The Edge Function URL is read from SUPABASE_EDGE_FUNCTION_URL env var.
This ensures progression happens even if the host disconnects.

### On client mount / reconnect:
Every time the player page mounts or the browser regains focus
(visibilitychange event), the client MUST call:
  GET /api/session/[sessionId]/state
which returns:
  {
    status: 'waiting' | 'active' | 'ended',
    currentQuestionIndex: number,
    questionStartedAt: string | null,     // ISO timestamp
    timeLimitSeconds: number | null,
    alreadyAnswered: boolean
  }

The client then:
- Reconstructs the timer from questionStartedAt
- Skips directly to the correct question
- If alreadyAnswered: show locked "answer submitted" state for remaining time
- If status is 'ended': redirect to results

- On mount: fetch session state from /api/session/[sessionId]/state
- If status is 'waiting': show waiting room with live player list
- If status is 'active': skip waiting room, sync directly to current
  question and server timer
- If status is 'ended': redirect to results
- Nickname input (default: random adjective+animal string)
- "Ready" button → insert player row, subscribe to realtime channel

Waiting room:
- Show "Waiting for host to start..."
- Player list updates in real time as others join via Postgres Changes

Gameplay (same route, state-driven):
- On quiz_start, next_question broadcast, or on mount sync:
  show current question with synced timer
- Timer bar width = (remaining / total) * 100%, updated every tick via
  requestAnimationFrame, derived from server questionStartedAt (not a
  local setInterval starting from full time)
- Answer buttons: tap/click to select (highlighted immediately)
- No submit button — selection is stored in local state only
- When client-side derived timer hits 0:
  - Auto-submit selected answer (or null if none selected) to answers table
  - Show "Waiting for next question..." overlay
  - Null submissions are marked as incorrect
- Answer cannot be changed after submission
- Progression to next question triggered only by next_question broadcast

### Explicitly removed features:
- No "Submit" / "Send" button
- No manual answer sending before timer ends
- No manual "Next Question" button anywhere

## DATABASE SCHEMA
Generate full SQL including:

Tables:
- users (extends auth.users via foreign key)
- invites (code, created_by, used_by, used_at, expires_at)
- quizzes (id, owner_id, title, description, created_at, updated_at)
- questions (id, quiz_id, text, answers jsonb, correct_index,
  time_limit_seconds, order_index)
- quiz_sessions (id, quiz_id, status, current_question_index,
  question_started_at, created_at, ended_at)
- players (id, session_id, user_id nullable, nickname, joined_at)
- answers (id, player_id, session_id, question_id, selected_index nullable,
  is_correct, response_time_ms, submitted_at)

Requirements:
- All foreign keys with ON DELETE CASCADE where appropriate
- created_at / updated_at on every table
- Indexes on: quiz_sessions.quiz_id, players.session_id, answers.player_id,
  answers.question_id, quiz_sessions.status + question_started_at
  (for the Edge Function scheduler query)
- RLS policies:
  - Only quiz owner can INSERT/UPDATE/DELETE on quizzes and questions
  - Only session players can INSERT on answers
  - Players can only insert their own answer (one per question per player)
  - Invite codes readable only by their creator or during registration flow
  - quiz_sessions readable by quiz owner and active players
